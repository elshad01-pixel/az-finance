## ⚠️ Kritik İnfrastruktur Problemi

Test sessiyasının ən başlanğıcında verilənlər bazasına (Supabase) hər müraciətdə eyni xəta qayıtdı:

```
{"error":"Invalid API key","hint":"Double check your Supabase `anon` or `service_role` API key."}
```

Bu xəta `db_select`, `db_insert` və bütün digər DB əməliyyatlarında təkrarlandı. Beləliklə, **4 ssenarinin heç birində verilənlər bazası ilə real qarşılıqlı əlaqə qura bilmədim** — nə faktura yarada, nə xərc daxil edə, nə də işçi/payroll məlumatlarını oxuya bildim. Bu, tətbiqin biznes məntiqinin problemi deyil — mühit/konfiqurasiya problemidir (yanlış və ya vaxtı keçmiş API açarı). Lakin son istifadəçi mühasib üçün bu, **tətbiqi tamamilə istifadə olunmaz vəziyyətə gətirir**.

Bununla yanaşı, **hesablama alətləri (`verify_payroll`, `verify_simplified_tax`, `summarise_financials`) işlədi** və onları sınaqdan keçirdim. Aşağıdakı qiymətləndirmələr həmin alətlərin nəticələrinə əsaslanır.

## Ümumi Qiymətləndirmə

AzFinance-in **hesablama nüvəsi (calculation engine) Azərbaycan vergi qanunvericiliyinə çox yüksək keyfiyyətlə uyğunlaşdırılmışdır** — PGV-nin proqressiv dərəcəsi, 200 AZN güzəşt, 75% sadələşdirilmiş vergi güzəşti və ƏDV həddinin izlənməsi düzgün işləyir. Bu, məhsulun ən güclü tərəfidir və rəqiblərlə müqayisədə üstünlükdür.

Lakin bu test sessiyasında **verilənlər bazası qatı tamamilə qoşulmaz idi** ("Invalid API key" xətası bütün əməliyyatlarda). Bu, P0 prioritetli infrastruktur problemidir və real mühitdə baş versəydi, məhsul saatlarla istifadə olunmaz qalardı. Daha ciddi məsələ odur ki, `summarise_financials` aləti DB səhvi olduqda **xəta atmaq əvəzinə sakitcə sıfır qaytardı** — bu, mühasib üçün ən təhlükəli davranışdır, çünki səhv qərarlara aparıb çıxara bilər.

Məsləhət: hesablama məntiqi istehsala hazırdır, lakin **infrastruktur dayanıqlığı və xəta idarəetməsi mütləq gücləndirilməlidir**. Bu iki şərt ödənilmədikcə tətbiqi peşəkar mühasiblərə tövsiyə edə bilmərəm.

## Prioritet Tövsiyələr

1. **[P0 — Kritik]** Supabase API açarı konfiqurasiyası dərhal düzəldilməlidir. Açar fırlatma (key rotation) prosesi və monitorinq qurulmalıdır ki, gələcəkdə açar müddəti bitdikdə avtomatik xəbərdarlıq gəlsin.

2. **[P0 — Kritik]** `summarise_financials` və digər oxuma alətləri DB xətası olduqda **mütləq açıq xəta qaytarmalıdır**, "0" yox. Bu səssiz uğursuzluq mühasib üçün maliyyə hesabatlarında səhv qərara səbəb ola bilər və auditi pozar.

3. **[P1 — Yüksək]** Bütün DB yazma əməliyyatları üçün **audit izi (audit log)** cədvəli olmalıdır: kim, nə vaxt, hansı qeydi dəyişdi, köhnə və yeni dəyər nə idi. Vergi orqanı yoxlamalarında bu sübut tələb olunur.

4. **[P1 — Yüksək]** Vergi hesablamalarının nəticəsi ilə birgə **qanun istinadı və düstur** göstərilməlidir (məs. "Vergi Məcəlləsi m.101.1, 14% dərəcə"). Bu, həm istifadəçi etibarını artırır, həm də auditdə müdafiə üçün vacibdir.

5. **[P2 — Orta]** Recurring xərclər (icarə, kommunal) və payroll-un xərc kateqoriyasına avtomatik ötürülməsi mexanizmləri əlavə edilməlidir — bu, ikiqat daxiletmə xətalarını azaldar.

6. **[P2 — Orta]** ƏDV həddi (500,000 AZN) yaxınlaşdıqda dashboard-da xəbərdarlıq və rüblük vergi ödəniş tarixləri üçün xatırlatma sistemi qurulmalıdır.
